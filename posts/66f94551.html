git与gihub的日常

发表于2026-02-22|更新于2026-02-22|工具
|总字数:4.4k|阅读时长:13分钟|浏览量:

前两篇做了什么

上两篇，我们完成了：

  • 安装 Git，配置身份
  • 注册 GitHub，创建仓库，把本地文件推送上去
  • 学会了 git add / commit / push，知道了怎么回溯版本

你现在已经可以独立管理自己的文件版本了。

但如果你去看真实的开发团队是怎么用 Git 的，你会发现他们做的事情比这多得多。他们会刻意忽略某些文件不提交，会从 GitHub 上直接拿别人的代码来用，会开分支做新功能，会把别人的修改同步回自己的电脑，会在合并代码前互相检查对方的工作。

这些，才是 Git 真正的日常。

.gitignore —— 哪些文件不该被提交

先想一个问题

你在做一个前端项目，node_modules 文件夹里装着几万个依赖文件，加起来可能有几百 MB。你需要把它推送到 GitHub 吗？

当然不需要。node_modules 里的东西任何人都可以通过 npm install 一键还原，把它推上去只会让你的仓库臃肿不堪，毫无意义。

除了 node_modules，还有很多文件不应该被提交：

  • 包含数据库密码、API 密钥的配置文件（.env）
  • 系统自动生成的缓存文件（.DS_Store、Thumbs.db）
  • 编译产物（/dist、/build）
  • 编辑器的私有配置（.idea/、.vscode/）

如果这些文件被提交上去，轻则仓库混乱，重则密钥泄露，你的服务器可能第二天就被人拿去挖矿了。

.gitignore 是什么

.gitignore 是一个放在项目根目录的文本文件，里面写的是你告诉 Git 要永远忽略的文件或文件夹。被忽略的文件，git add . 时不会暂存，git status 时不会出现，就像它们不存在一样。

怎么写

在你的项目根目录创建一个名为 .gitignore 的文件（注意前面有个点），用任何编辑器打开，按规则填写：

规则很直觉：写文件夹名加 / 表示忽略整个文件夹，写 *.xxx 表示忽略所有这个后缀的文件。

不用从零开始写

GitHub 维护了一个针对各种语言和框架的 .gitignore 模板库：

https://github.com/github/gitignore

里面有 Python、Node、Java、Vue、React 等几十种常用模板。你做什么项目，就复制对应的模板，贴进你的 .gitignore 就好。

另外，在 GitHub 创建仓库时，页面下方有一个 Add .gitignore 的选项，选好语言后会自动帮你生成：

⚠️ 已经被追踪的文件，加进 .gitignore 不会生效。 如果你之前已经 commit 过某个文件，后来才加进 .gitignore，Git 依然会继续追踪它。需要先用 git rm --cached 文件名 把它从追踪列表里移除，再 commit 一次，才会真正被忽略。

git clone —— 把别人的项目拿到本地

开源的意义在于你能真正用它

前面提到，GitHub 上有几亿个公开仓库。那些代码不是摆设，你可以直接把它们下载到本地运行、学习、甚至在它的基础上继续开发。

这个动作叫 clone（克隆）。

clone 下来的项目，是一个完整的 Git 仓库，所有的 commit 历史都在，你可以查看，可以回溯，可以在本地做任何修改。

clone 和下载 ZIP 的区别

GitHub 页面上有一个 Download ZIP 的选项，很多人会用这个。但它和 clone 有本质区别，而且这个区别比表面看起来更深。

Git 仓库本质上是一条时间维度上的快照链——每一个 commit 都是项目在某个时刻的完整状态，所有快照首尾相连，形成一段可以任意回溯的历史。当你 git clone，你拿到的是整段影像；当你 Download ZIP，你拿到的只是某一帧截图。

没有历史，你就无法知道某行代码是谁在什么时候为什么写的，无法回滚，无法跟着项目更新，也无法向项目贡献代码。

对比维度 git clone Download ZIP
历史记录​ ✅ 完整保留 ❌ 没有
后续更新​ ✅ 可以 git pull 同步 ❌ 需要重新下载
可以提交贡献​ ✅ 可以 ❌ 不可以

如果你只是想快速看看某个项目的代码，Download ZIP 够用。但如果你打算认真学习、跟着更新、或者参与贡献，永远用 clone。

一个真实的场景

分支 —— Git 最强大的武器

为什么需要分支

假设你的博客已经在线上稳定运行了，现在你想加一个新功能。你直接在主线上改，改到一半——产品经理说这个方向不对，不要了。

你怎么办？回滚？那你这几天的工作全没了。

或者，你和另一个人在合作同一个项目，你在写登录功能，他在修复一个 Bug，你们都在改同一份代码，每次保存都会覆盖对方的工作。

这些问题，分支全部解决了。

分支是什么

分支就是从主线上”开叉”出去的一条独立时间线。你在分支上做的所有改动，不会影响主线，也不会影响其他分支。等你做完了、确认没问题了，再把它合并回来。

用图理解：

这意味着：

  • 主线（main）永远是稳定可用的版本
  • 新功能、修复、实验，都在独立的分支上进行
  • 多人协作时，每个人有自己的分支，互不干扰

这里有一点初学者容易误解：分支创建时，内容和原分支完全一样，不是一个空文件夹。你从 main 创建 feature-login，原来的所有文件都会出现在里面——这是正常的。分支是从某个时间点”分叉”出去的平行时间线，起点相同，之后各走各的路，互不影响。

遇到冲突怎么办

如果你和另一个人改了同一个文件的同一个地方，合并时 Git 无法自动决定保留哪个，就会发生冲突（conflict）。

理解冲突的本质很重要：它发生的原因是两条平行时间线都改写了同一个节点——从同一个起点分叉，各自走出了不同的历史，现在要汇合，Git 不知道哪一边的世界线才是”最终答案”，所以它停下来，把决定权还给你。

Git 会在冲突的文件里插入这样的标记：

你需要手动打开文件，决定保留哪一段（或者两段都要），删掉那些标记符号，再重新 add 和 commit。

冲突听起来可怕，其实解决起来很直觉——你就是一个裁判，看看两边改了什么，决定最终要什么。而且理解了冲突的成因，你自然知道如何预防：分支存活时间越短、合并越频繁，冲突就越少——平行时间线分叉越短，汇合时的差异就越小。

大多数编辑器（VS Code、WebStorm）都有可视化的冲突解决界面，点击”保留这个”或”两个都要”就能解决，不需要手动编辑那些标记符号。

Pull Request —— 合并之前，先让人看看

为什么不直接 merge

上面讲的 git merge 是你在本地自己合并自己的分支，没人检查，合了就合了。

但在真实的团队里，没有人会允许你把代码直接合进主线。原因很简单：你的代码可能有 Bug，可能和别人的改动冲突，可能不符合团队的规范。代码在合并之前，需要有人检查。

这个”合并申请 + 代码审查”的机制，在 GitHub 上叫做 Pull Request（简称 PR）。

名字听起来奇怪，理解起来其实很直接：你在说”我请求你把我的分支拉（pull）进你的主线”。

Pull Request 是什么

PR 是 GitHub 提供的一个界面，它做三件事：

  1. 展示差异：让所有人清楚地看到这个分支相比主线改了什么
  2. 讨论区：队友可以在具体的代码行上留下评论，提出修改意见
  3. 合并入口：review 通过后，点一个按钮完成合并

整个流程，代码的每一次变动都有迹可查，每一次合并都经过审核。这就是为什么几十个人协作的开源项目能井然有序地运转。

怎么发起一个 Pull Request

第一步：把你的分支推上 GitHub

第二步：在 GitHub 上创建 PR

推送完成后，打开 GitHub 仓库页面，你会看到一个黄色提示栏，提示你刚刚推送了新分支，并提供一个 Compare & pull request 按钮，点它：

如果没有看到提示栏，也可以手动点击 Pull requests 选项卡 → New pull request，选好”从哪个分支”合并”到哪个分支”。

第三步：填写 PR 信息

这一步很多人敷衍了事，但它其实很重要。一个好的 PR 描述，应该让审查者不需要运行你的代码，就能理解你做了什么、为什么这么做：

  • 标题：简洁说明这个 PR 做了什么，比如”添加用户登录功能”
  • 描述：这个改动的背景是什么，做了哪些具体的事，有没有需要特别注意的地方
  • Reviewers：指定你希望谁来审查（右侧栏）

填好后点击 Create pull request。

代码审查是怎么进行的

PR 创建后，被指定的 reviewer 会收到通知。他打开 PR 页面，点击 Files changed 选项卡，能看到这次改动的每一行：红色是删除，绿色是新增。

他可以在任意一行代码上悬停，点击出现的 + 号，留下评论。你会收到通知，回来看评论、修改代码、再次提交——同一个 PR 可以持续追加新的 commit，不需要关掉重开。

所有问题解决后，reviewer 点击 Review changes → Approve，表示代码通过审查。

合并 PR

审查通过后，来到 PR 页面底部，点击 Merge pull request 按钮，GitHub 会自动把这个分支合并进主线，并在远程创建一条合并 commit。

合并后，GitHub 会提示你删除这个已完成的分支，点击即可。

Fork + Pull Request：给陌生人的项目贡献代码

上面讲的是在同一个仓库内的协作。但如果你想给一个你没有写权限的开源项目贡献代码，流程会稍有不同，需要用到 Fork。

Fork 就是把别人的仓库完整地复制一份到你自己的 GitHub 账号下。复制出来的仓库完全归你所有，你可以随意改动，不会影响原仓库。

流程是这样的：

打开任意一个 GitHub 公开仓库，右上角有一个 Fork 按钮，点击后会在你的账号下生成一个副本。之后的操作和普通 PR 完全一样——clone 你自己的 Fork，新建分支，修改，push，发 PR，只是 PR 的目标从”同仓库的另一个分支”变成了”原仓库的主分支”。

这就是 GitHub 上几乎所有开源项目的协作方式。你在用的很多库，它的 contributors 列表里每一个人，都是通过这个流程贡献了自己的代码。

几个日常高频命令速查

命令 作用
git status 查看当前哪些文件被修改了、暂存了
git log --oneline 简洁查看提交历史
git diff 查看具体改了什么内容
git stash 临时”藏起”当前未提交的改动（切换分支前很有用）
git stash pop 把藏起来的改动恢复回来
git branch -a 查看所有分支，包括远程的
git remote -v 查看当前仓库连接的远程地址

结语

有一个画面，是很多人真正”入门” Git 的时刻：

你在一个知名开源项目里发现了一个 Bug，查了半天，确认是代码问题。你 Fork 了仓库，建了一个分支，改了五行代码，写了一个 PR，说明了问题和你的解法。两天后，一个你从未见过的维护者在你的 PR 下回复了一句”LGTM”——Looks Good To Me——然后点了合并。

从那一刻起，全世界用这个库的人，用的都是含有你那五行代码的版本。

这就是 Git 和 GitHub 给你的能力。工具本身并不复杂，复杂的是人与人之间协作所需要的信任机制——而 PR，就是这个机制的具体形式。
