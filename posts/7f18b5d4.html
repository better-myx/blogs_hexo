github理解与实操

发表于2026-02-22|更新于2026-02-22|工具

先问你一个问题

你有没有想过，Linux 这个操作系统是怎么做出来的？

它运行在全球超过 96% 的服务器上，支撑着你每天用的微信、淘宝、抖音背后的基础设施。但它不是某个公司做的，不是某个团队做的——它是由来自全球数万名素不相识的程序员，用几十年的时间，一行一行堆出来的。

他们从未开过一次全员会议。他们可能说着不同的语言，住在不同的时区，甚至从未见过彼此。

他们是怎么做到的？

答案的一半是 Git。另一半，就是今天要讲的 GitHub。

上一篇留下了一个问题

上一篇我们装好了 Git，它能追踪你文件的每一次改动，让你随时回溯。听起来很强大——但有一个问题你可能还没想到：

这些记录，存在哪里？

答案是：存在你自己的电脑上。

你的硬盘坏了，记录没了。你换了一台电脑，记录没了。你想让别人看你的内容，他们看不到。

所以光有 Git 还不够。你需要一个地方，把这些版本记录同步到云端，随时可以访问，永远不会丢。

这个地方，就是 GitHub。

GitHub 是什么？——别被它的名字骗了

很多人第一次听到 GitHub，以为它就是”Git 的官网”或者”一个存代码的网盘”。

这个理解，就像把图书馆理解成”一个放书的仓库”——技术上没错，但你错过了它真正的意义。

GitHub 的本质是：一个以代码为载体的协作平台。

  • Git 是你桌子上的时间机器（本地）
  • GitHub 是把这台时间机器接入了全球互联网（云端 + 社区）

截至 2024 年，GitHub 上有超过 4.2 亿个代码仓库，注册开发者超过 1 亿人。你现在每天用的很多软件，背后的代码就托管在这里：VS Code、React、Python、甚至 Git 本身，都在 GitHub 上公开。你可以直接去看，如果发现了 Bug，甚至可以提交修复，被全世界的人用到。

这件事放在 20 年前是不可想象的。GitHub 让代码变成了一种可以公开协作的知识。

过去，代码是私有财产，锁在公司服务器里，外人看不见，更别说参与。GitHub 出现之后，一个人在宿舍里写的小工具，推送上去，第二天可能有来自另一个国家的人 clone 了它，修了一个 Bug，提交了 Pull Request，一年后被几万人使用。

这不是故事，这是 GitHub 上每天都在发生的事。

不过今天我们先不聊这些宏大的事。我们只做一件具体的事：把你本地的文件，推送到 GitHub 上。

第一步：注册 GitHub 账号

访问 https://github.com，点击右上角 Sign up。

填写邮箱、密码、用户名，完成邮箱验证即可。

用户名怎么取？ 它会出现在你所有仓库的链接里：github.com/你的用户名。 相当于你在开发者社区的门牌号，建议用英文，简洁，和你的技术身份有关联，认真对待它。

第二步：创建一个远程仓库

登录后，点击右上角 “+” → New repository。

填写仓库信息：

字段 填写建议
Repository name 英文小写+连字符，如 demo、my-first-repo
Description 可选，简单描述用途
Public / Private 学习阶段选 Public 即可
Add a README file 先不勾选，保持空仓库，这样进入页面后它会直接给你连接仓库的命令

点击 Create repository，你的云端仓库就诞生了。

逐行读懂这段命令

git init —— 在这个文件夹里启动 Git

这行命令告诉 Git：把这个文件夹变成一个 Git 仓库，从现在开始追踪里面的变化。

执行完之后，文件夹里会多一个隐藏的 .git 文件夹——那是 Git 的”大脑”，所有版本记录都存在里面。你不需要动它，知道它在就好。

每个项目只需要执行一次，之后不用再 init 了。

git add —— 告诉 Git 你要提交哪些文件

Git 不会自动帮你记录所有改动。你必须明确告诉它：我要把哪些文件纳入这次提交。

这一步叫”暂存”——就像你整理东西准备打包，先把要装箱的挑出来放在一边。

. 的意思是”把当前文件夹里所有改动过的文件全部暂存”。实际使用中你几乎永远会用这个，一个点，省掉所有麻烦。

git commit —— 正式提交，拍下这一刻的快照

commit 是 Git 的核心动作。执行之后，Git 会把你刚才 add 的内容永久记录下来，打上时间戳和你的名字，生成一个唯一的版本 ID。

-m 后面的引号里是提交信息，用来描述这次改动做了什么。

提交信息怎么写？把它当成写给三个月后的自己看的备忘录。

  • ✅ "完成第三章初稿"
  • ✅ "修改结论部分，补充数据来源"
  • ❌ "改了点东西"
  • ❌ "aaa"

你以后翻历史记录的时候，会感谢现在认真写备注的自己。

git branch -M main —— 把当前分支命名为 main

Git 里有”分支”的概念，你可以把它理解为”平行时空”——同一份文件，可以同时存在多个分支，互不干扰，最后再合并。

这行命令只是把当前这条时间线命名为 main，也就是主线。这是 GitHub 的默认规范，跟着用就好。

分支是 Git 最强大的功能之一，我们后面会专门讲。现在只需要知道：有这行命令，是为了和 GitHub 的命名规范保持一致。

git remote add origin —— 告诉本地 Git，云端仓库在哪里

目前为止，你的本地 Git 和 GitHub 完全没有联系，它们是两座孤岛。

这行命令就是在架桥——告诉本地 Git：有一个远程仓库，地址是这个，我给它起个别名叫 origin。

origin 只是别名，约定俗成这么叫，跟着用就好。

git push —— 把本地提交推送到云端

这是最后一步。push 就是”推送”——把你本地所有的 commit，推送到 origin（GitHub）的 main 分支上。

-u 是第一次推送时用的参数，它会记住这次的推送目标。之后你每次推送，直接输 git push 就够了，不需要再加 -u origin main。

HTTPS 和 SSH：两种握手方式

回到上面那段命令，你会注意到 GitHub 页面上有 HTTPS 和 SSH 两个选项卡可以切换：

它们的区别只在 git remote add origin 那行的链接格式，命令其余部分完全一样。但背后是两套完全不同的身份验证机制。

方式一：HTTPS（开箱即用）

GitHub 曾经允许用账号密码验证，但 2021 年关掉了这个方式。现在 HTTPS 背后用的是 Personal Access Token（个人访问令牌），本质上是 GitHub 给你生成的一串专用密码。

但你大概率从来没见过它——因为 Git for Windows 安装时自带了一个叫 Git Credential Manager 的工具，第一次 push 时它会悄悄在后台完成 GitHub 授权，把凭据缓存进 Windows 凭据管理器，之后就永远不出现了。

这就是为什么很多人用 HTTPS 从始至终都没输过密码——不是没有验证，是系统替你做了。

方式二：SSH（一次配置，永久免验证）

SSH 不依赖任何密码或 Token，用的是密钥对验证。

你在本地生成两把密钥：公钥和私钥。把公钥上传给 GitHub，私钥永远留在自己电脑里。之后每次 push，双方自动完成验证，全程无感，不依赖任何系统缓存。

用生活类比：公钥是你家的门锁，私钥是你手里的钥匙。锁给了 GitHub，钥匙永远在自己手里，即使 GitHub 服务器被攻击，你的私钥也毫发无损。

先去逛逛，感受一下这个地方

在继续之前，我想让你做一件事：

打开 GitHub，搜索任何你感兴趣的关键词——python、game、blog，什么都行。

点进任何一个仓库，随便翻翻：看看 README 写了什么，看看有多少人在用，看看别人在 Issues 里讨论什么问题。

不需要看懂，只需要感受到这个地方的体量。

这是全世界开发者公开展示自己工作的地方，也是你以后会越来越频繁回来的地方。

结语

你现在已经走通了一个完整的链路：本地文件 → Git 追踪 → 推送到 GitHub → 随时可以回溯。

不管你现在放上去的是论文、笔记还是代码，这套流程是一样的。

但这还只是你一个人在用 Git。而 Git 和 GitHub 真正强大的地方，是当多个人围绕同一份代码协作的时候——分支、合并、Pull Request、冲突解决。
